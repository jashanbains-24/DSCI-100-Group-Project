Project Planning Stage

(1) Data Description:

The two files given to us to use are "players.csv" and "sessions.csv". They have 196 and 1535 obsevations, respectively. The players dataset contains information about demographics and engagement data whereas the sessions dataset tracks gameplay sessions.

There are some key issues with both of these datasets that we will need to deal with before starting our project. For example, it should be noted that play time distribution is quite skewed (most players have minimal play time while only a few exceed 200 hours).

For sessions.csv, the two "original_start_time" and "original_end_time" both provide the same information as "start_time" and "end_time". We should choose to one for consistency.

There also appears to be missing values in both datasets:

  • Players data has two missing values in the age column (N/A)
  • Sessions data has two missing "end_time" values

Players Dataset (players.csv)

  • experience (chr):Player type (Pro, Veteran, Regular, Amateur)
  • subscribe (lgl): A boolean representing whether they are subscribed
  • hashedEmail(chr): Identifier for linking players across datasets
  • played_hours(dbl): Total hours the player has spent in the game
  • name(chr): Actual name (not gamertag)
  • gender(chr): Players gender
  • age(dbl): Players age in years

Sessions Dataset (sessions.csv)

  • hashedEmail(chr): Identifier linking players across datasets
  • start_time (chr): Session start time (untidy)
  • end_time(chr): Sessions end time (untidy)
  • original_start_time(dbl): Unix time representation (not useful for us)
  • original_end_time(dbl): Unix time representation (not useful for us)

I have included the summary statistics as a part of my exploratory section

(2) Questions:

The broad question I aim to address is "Which kinds of players are most likely to to contribute a large amount of data?"

To explore this, I have formulated the more specific question: "Can we predict a player's type based on their in-game behaviour, such as time spend playing, newsletter subscription, and sessions.

To answer this question we use experience as the response variable, and played_hours, subscribe, and session_metrics which will be created through wrangling and mutating the data.

The datasets will allow me to address this question due to many reasons:

  • Provide key player information and engagement metrics
  • Allow me to determine patterns between variables
  • Merge datasets and create a more comprehensive observation for each player

Possible Wrangling Steps:

  • Merging datastes to utilize hashedEmail
  • Creating variables from session data that describe session length with one column instead of start and end time

(4) Methods and Plan

To predict player type based on engagement metrics, we will use K-Nearest Neighbors Classification. This model is the right choice for this analysis as it classifies player experience by comparing them to other entries with similar features, using played hours, sessions, and age.

Assumptions

  • Similar Players need to exist in the dataset, will be needed for comparability for training our mdodel
  • The straight line distance will reflect players accurately after scaling and centering to ensure fair classification

Limitations

  • One limitation is that this model can be fairly expensive in terms of computation as the dataset grows, and we are dealing with a large dataset. To battle this, it is crucial that we choose the right K through cross validation. This will optimize K and prevent over/underfitting.
  • Another weakness is that there are some outliers in this dataset and if we choose the wrong K value, those might influence our classfication negatively (we need to ensure that we have the right K value)

Comparing the Model

  • We will evaluate K values using cross-validation(5-10 fold) and compare using accuracy, precision and recall. This will ensure our model generalizes well and does not overfit to training data. We will then do the same with linear regression and comparing rmspe values to ensure that we have made the right choice going with classification.

  • We will be splitting the data using an 80/20 training and testing split. We will apply 5 fold cross-validation. Then finally, we will use resulting value of K to make final predictions.
